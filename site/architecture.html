Architecture

Two current execution paths: metrics and replay.

Lighthouse is still early in the rebuild, so the architecture is kept intentionally direct: a read-only metrics path for cluster visibility and a CLI path for replay execution.

Dashboard Flow

Browser
  -> Next.js page
  -> /api/dashboard-metrics
  -> Prometheus fetch client
  -> Prometheus HTTP API
  -> Kafka metrics exporter
  -> Kafka Admin metadata

Prometheus access stays on the server side. The browser never hits Prometheus directly.

Replay Flow

CLI input
  -> Kafka admin validation
  -> Source topic subscription
  -> Partition seek to start offset
  -> Per-message preview or replay
  -> Destination topic write
  -> Summary and progress logs

Local Sample Stack

  • Three Kafka brokers in KRaft mode
  • Topic initializer for orders, payments, and orders-replay
  • Demo producer for recurring sample traffic
  • Kafka metrics exporter
  • Prometheus
  • Next.js application

Why the Rebuild Uses This Shape

  • It keeps production Prometheus and Kafka access behind explicit configuration.
  • It proves replay behavior before adding job orchestration and UI complexity.
  • It gives the project a stable demo story that does not depend on hosted clusters.

The next architectural step is a persistent replay job model with a small API layer over the engine.
